# Claude Code コスト削減プラン

## 要求

Claudeのコスト削減を行いたい。

- 作業のsonnet, haikuへの委任が足りていない
- コマンドについて、軽量モデルでも品質にばらつきのでなさそうなものをリストアップ
- planの実行自体についてベストプラクティスはあるだろうか？
  - 大前提としてOpusより精度が落ちるのは許さない
  - "OpusがSonnet/Haikuが確実に失敗しない分割タスクを見出し委任。それがより安い"であれば、行うように
  - 一般的なベストプラクティスをリストアップ
- その他ツールなど

## 調査結果

### 1. 現在のモデル委任状況

22個のカスタムコマンドのうち、軽量モデルに委任しているのは **3個のみ**（すべてHaiku）:

| コマンド | モデル | 委任方式 |
|---|---|---|
| `/org-agenda` | haiku | Task subagent 全委譲 |
| `/org-bump` | haiku | Task subagent 全委譲 |
| `/org-prune` | haiku | Task subagent 全委譲 |

残り19コマンドはメインセッションモデル（通常Opus）で実行。

### 2. コマンド分析: 委任可能性

各コマンドを「タスクの性質」と「インタラクティブ性」で分類した。

#### Tier A: Haiku委任可能（純粋に機械的）

| コマンド | 理由 |
|---|---|
| `/show-idle-agenda` | emacsclient -e を3回実行するだけ。分析は窓面積の比較のみ |

#### Tier B: Sonnet委任可能（手続的・パターンベース）

| コマンド | 理由 | インタラクション |
|---|---|---|
| `/nix-check` | コマンド実行→パターン照合→レポート生成。読み取り専用 | なし |
| `/format` | フォーマッタ実行→diff表示のみ | フォーマッタ未検出時のみ |
| `/changelog` | git log分類→Markdown生成。パターンマッチング主体 | CHANGELOG.md保存確認のみ |
| `/cleanup` | git branch状態確認→リスト提示 | 各削除に個別確認あり |
| `/dep-update` | 更新コマンド実行→diff表示 | 更新対象の選択 |
| `/commit` | diff分析→スタイル検出→メッセージ生成。Sonnetの要約能力は十分 | メッセージ確認 |
| `/pr` | コミット分析→PR本文生成。要約タスク | タイトル/本文確認 |

#### Tier C: Sonnet委任は条件付き

#### Tier D: Opus維持（複雑な推論・コンテキスト依存）

| コマンド | 理由 |

| `/team` | オーケストレーション。ただしワーカーモデル指定は改善可能 |

| `/init-work` | 同上 |
| `/worktree` | 複雑なgit操作＋環境構築 |
| `/org-todo` | 会話コンテキストが必要（subagentに委任不可） |

### 3. subagent委譲パターン（既存実装）

org-agenda等で使用されているパターン:

```markdown
Task ツールで以下の通り subagent を起動し、全処理を委譲せよ:
- subagent_type: `"general-purpose"`
- model: `"haiku"`
- prompt: <詳細プロンプト>

subagent の出力をそのままユーザーに表示すること。
```

Tier A/Bのコマンドにこのパターンを適用する（modelをhaiku/sonnetに設定）。

### 4. Plan実行のベストプラクティス

#### a. /team ワーカーモデル最適化

現在 `/team` はワーカーモデルを指定していない（親のOpusを継承）。

改善案:
- `/team` コマンドにワーカーデフォルトモデルの指定を追加
- プランの各タスクに複雑度を明示し、Opus必須タスクとSonnet可タスクを区別
- 実装系の定型タスク（テスト記述、定型コード生成等）→ Sonnetワーカー
- 設計判断が必要なタスク → Opusワーカー

#### b. Opusによるタスク分解→Sonnet実行パターン

1. **明確な入出力定義**: Sonnetに委任するタスクは「入力・期待出力・制約」を厳密に指定
2. **例示の付与**: プロンプトに具体例を含める（Few-shot）。軽量モデルほど例示が効く
3. **検証ループ**: Sonnet実行結果をOpusが検証し、不十分なら再実行またはOpusが修正
4. **タスク粒度**: 1タスク=1明確なゴール。曖昧なタスクはOpusで分割してから委任

#### c. 一般原則

- **機械的タスクの識別**: 手順が明確で判断が不要なタスクは常にSonnet/Haiku候補
- **品質ゲート**: 委任後の出力をOpusが最低限レビューする（コスト < Opus全実行）
- **段階的移行**: まずTier B上位（nix-check, format, changelog）で試行→品質確認→拡大

### 5. その他のコスト最適化

#### a. コンテキスト管理（最大効果）

- **`/clear`の活用**: 無関係なタスク間でコンテキストをリセット
- **CLAUDE.mdサイズ**: 現在のCLAUDE.mdは全セッションで読み込まれる。専門的な内容はスキルに移動してオンデマンド化を検討
- **compaction指示**: `/compact <focus>` でコンテキスト圧縮時の重点を指定

#### b. MCP・ツール定義の最適化

- 各MCPサーバーのツール定義がコンテキストに追加される
- 未使用サーバーは `/mcp` で無効化
- CLIツール（gh, aws等）が利用可能な場合はMCPより優先

#### c. セッション管理

- 短い多数のセッションより、長い少数のセッションが効率的（キャッシュ作成コストの償却）
- Agent Teamsセッションは特にコストが高い（ワーカー数に比例。3人で3-4倍）

#### d. Extended Thinking制御

- Opus 4.6のデフォルト思考バジェット: 31,999トークン
- 単純タスクでは `/model` のeffortスライダーで低減可能
- 環境変数 `MAX_THINKING_TOKENS` で制御

#### e. Prompt Caching

- Claude Codeは自動的にプロンプトキャッシングを使用（デフォルト有効）
- システムプロンプトとCLAUDE.mdがキャッシュ対象

### 6. コスト構造の分析

ccusageデータの主要コスト要因:

| 要因 | トークン数 | 備考 |
|---|---|---|
| Cache Create | 85M | Opusの場合高コスト。セッション開始・コンテキスト変更で発生 |
| Cache Read | 557M | キャッシングにより低コスト |
| Input | 3M | 比較的少量 |
| Output | 1.6M | 比較的少量 |

→ **Cache Createの削減**（セッション数を減らす、CLAUDE.mdを小さくする）と **Opusトークンの削減**（Sonnet/Haiku委任）が主な最適化ポイント。
