# /explore - コードベース探索・解説

コードベースの構造や特定コンポーネントのアーキテクチャを調査・解説する。

**重要: このコマンドは読み取り専用。ファイルの作成・変更・削除は一切行わない。**

## 引数

$ARGUMENTS: 調査対象（省略時はプロジェクト全体の概要）

例:
- `/explore` - プロジェクト全体の概要
- `/explore 認証システム` - 認証関連のコードを調査
- `/explore src/api/` - 特定ディレクトリの構造を解説
- `/explore エラーハンドリング` - 横断的な関心事を調査

## 処理フロー

以下のステップを順に実行し、結果をまとめて報告する。

### 1. プロジェクト基本情報の収集

- ディレクトリ構造の把握（`ls`, `tree`等）
- エントリポイントの特定（main関数、index.ts、flake.nix等）
- README.mdがあれば内容を確認

### 2. 依存関係の分析

依存管理ファイルから外部依存を一覧化:
- `Cargo.toml` → Rust依存
- `package.json` → Node.js依存
- `pyproject.toml` / `requirements.txt` → Python依存
- `go.mod` → Go依存
- `devenv.nix` / `flake.nix` → Nix依存

### 3. アーキテクチャパターンの特定

コード構造から以下を特定:
- アーキテクチャパターン（MVC, Clean Architecture, Hexagonal等）
- レイヤー構成（presentation, domain, infrastructure等）
- 主要なデザインパターンの使用

### 4. 対象に応じた深堀り

$ARGUMENTSが指定されている場合:
- 関連するファイルとコードを検索
- データフローとコンポーネント間の依存を追跡
- 公開インターフェースと内部実装を区別

### 5. 報告

以下の構造で結果を報告する:

```
## 概要
<プロジェクトの目的と技術スタック>

## ディレクトリ構造
<主要ディレクトリとその役割>

## 主要コンポーネント
<核となるモジュール/クラス/関数の説明>

## 依存関係
<外部依存の一覧と用途>

## アーキテクチャ
<パターン、レイヤー構成、データフロー>

## 注目ポイント
<特筆すべき設計判断、改善の余地、注意点>
```

$ARGUMENTSが指定されている場合は、該当トピックに焦点を当てた構成にする。
